# ADR-0001: Pinchtab MCP Wrapper (pinchtab-mcp-wrapper)

Status: Proposed  
Date: 2026-02-19  
Owners: Platform / Agents Team

## 1) Цели и задачи

### Контекст
Нужно сделать MCP-обёртку над Pinchtab (HTTP browser server), чтобы использовать её как локальный browser-agent в среде agentic development (opencode).  
Среда: Ubuntu 22.04.5 LTS. На VM **нет установленного Chrome/Chromium**, поэтому Pinchtab должен запускаться так, чтобы браузер был доступен без установки системного Chrome.

### Цели
- Дать LLM-агентам полный набор возможностей Pinchtab через MCP (stdio, локально).
- Существенно снизить токен-нагрузку на "чтение" страниц и повторные проверки:
  - Prefer `/text` для чтения,
  - Prefer `/snapshot?filter=interactive&format=compact|text&diff=true` для действий и "проверок изменений",
  - Поддержать `maxTokens`, `depth`, `selector` и file-output сценарии.
- Обеспечить работу в режиме:
  - локально (в т.ч. на VDS), без внешнего доступа,
  - с конкурентным доступом (несколько агентов / асинхронные запросы),
  - с безопасной локальной эксплуатацией (token + loopback).
- Поддержать скриншоты двумя способами:
  1) inline base64 в ответе MCP tool,
  2) выгрузка в S3 (endpoint/bucket/credentials задаются в конфиге).

### Не-цели
- Не делаем "умного браузера" поверх Pinchtab (планирование, чтение мыслей агента и т.п.) — только надёжная MCP-обвязка + несколько удобных "макро" операций для токен-экономии.
- Не делаем удалённый (public) сервер; только local-first.

## 2) Решения

### 2.1 Transport и модель запуска
- MCP сервер: **stdio** (локальный агент).
- Pinchtab запускается и используется **только локально**:
  - Wrapper общается с Pinchtab по HTTP через `127.0.0.1` (или docker network).
  - В прод-режиме по умолчанию Pinchtab порт наружу не публикуем (или публикуем только на loopback).

### 2.2 Как обеспечиваем браузер без установленного Chrome/Chromium
Решение по умолчанию: **Pinchtab как sidecar Docker container** (внутри Chromium).  
Wrapper в режиме `PINCHTAB_MODE=auto`:
1) если задан `PINCHTAB_URL` → подключается к существующему Pinchtab;
2) иначе → поднимает docker-контейнер `pinchtab/pinchtab:<tag>` (по умолчанию pinned tag), ждёт `/health`, затем работает.

Причины:
- На VM нет Chrome/Chromium; Docker-образ Pinchtab уже содержит Chromium и нужные флаги.
- Меньше "дрейфа" окружения и меньше ручных шагов.

Альтернативы (не выбранные по умолчанию):
- Local binary Pinchtab + системный Chrome/Chromium (подходит для dev-машин, но не для заданной VM).
- "Два-в-одном" контейнер (wrapper+pinchtab в одном image) — возможно, но чаще избыточно; оставляем как опциональный publish.

### 2.3 Поверхностный дизайн MCP tools (полный набор + макро-обёртки)
Wrapper экспонирует два слоя:

#### A) Thin tools (1:1 с HTTP API Pinchtab)
- `pinchtab_health()`
- `tab_list()`
- `tab_open(url?: string)`
- `tab_close(tabId: string)`
- `navigate(tabId: string, url: string)`
- `snapshot(tabId?: string, filter?: "interactive", format?: "json"|"text"|"compact"|"yaml", diff?: boolean, selector?: string, depth?: number, maxTokens?: number, noAnimations?: boolean, output?: "inline"|"file", path?: string)`
- `text(tabId?: string, mode?: "readability"|"raw")`
- `action(tabId: string, kind: "click"|"type"|"fill"|"press"|"focus"|"hover"|"select"|"scroll"|"humanClick"|"humanType", ref?: string, selector?: string, text?: string, value?: string, key?: string, x?: number, y?: number, scrollX?: number, scrollY?: number)`
- `evaluate(tabId: string, js: string)`
- `tab_lock(tabId: string, owner: string, timeoutMs?: number)`
- `tab_unlock(tabId: string, owner: string)`
- `screenshot(tabId?: string, quality?: number, noAnimations?: boolean, delivery?: "base64"|"s3"|"file", path?: string)`

Примечания:
- `ref` предпочитаем как основной способ действий (стабильные `e0`, `e1`…).
- `selector` даём как "escape hatch" (Pinchtab тоже поддерживает часть действий по CSS).
- `humanClick/humanType` прокидываем (для антибот сценариев).

#### B) Token-saver macro tools (рекомендуемые для агентов)
- `read_page(tabId?: string, mode?: "readability"|"raw")` → вызывает `/text`.
- `list_interactives(tabId?: string, maxTokens?: number)` → `/snapshot?filter=interactive&format=compact`.
- `observe_changes(tabId?: string, filter?: "interactive"|null, format?: "compact"|"text", maxTokens?: number)` → `/snapshot?diff=true...`.
- `read_region(tabId: string, selector: string, format?: "text"|"compact", maxTokens?: number)` → `/snapshot?selector=...` (или evaluate как fallback).

### 2.4 Конкурентность и мульти-агентность
- Используем встроенный механизм Pinchtab: `/tab/lock` и `/tab/unlock`.
- Wrapper не "угадывает" owner: tool-параметр `owner` обязателен.
- Дополнительно внутри wrapper:
  - per-tab async mutex (на время одного HTTP-call) чтобы гарантировать корректный порядок для "сложных" макро-операций (например: snapshot → action → snapshot).
  - таймауты и отмена запросов: если MCP client отменил вызов, wrapper закрывает HTTP запрос (best effort).

### 2.5 Безопасность (local-only)
- Всегда включаем auth token на стороне Pinchtab:
  - либо берём из `PINCHTAB_TOKEN`,
  - либо генерируем при auto-start и держим только в памяти wrapper (не логируем).
- Пробрасываем token в `Authorization: Bearer <token>` в Pinchtab HTTP.
- Порт Pinchtab наружу по умолчанию не публикуем, либо публикуем только на `127.0.0.1`.
- State dir (профиль Chrome, cookies, tabs) лежит в volume и считается секретом.

### 2.6 Скриншоты: base64 и S3
- `delivery=base64`:
  - wrapper получает JPEG bytes от Pinchtab и возвращает base64 строкой + метаданные.
- `delivery=s3`:
  - wrapper получает JPEG bytes,
  - грузит в S3 (или S3-compatible, напр. MinIO),
  - возвращает URL + ключ.
- Если `delivery=file`: используем Pinchtab `output=file` (и/или пишем сами) и возвращаем path.

## 3) Автотесты и e2e-тесты

### 3.1 Unit tests (wrapper)
- Валидация схемы входов tools (required поля, mutually exclusive ref/selector и т.п.).
- Маппинг параметров tool → HTTP запрос Pinchtab (query/body/headers).
- Обработка ошибок:
  - Pinchtab недоступен,
  - 401 при неверном token,
  - 4xx при неверных параметрах,
  - timeouts.
- Скриншоты:
  - base64 корректно декодируется,
  - s3 uploader корректно формирует ключ/URL (mock).

### 3.2 Integration tests (wrapper + real Pinchtab)
Запускаем docker-compose:
- pinchtab container (с Chromium),
- wrapper (в test-mode),
- опционально: MinIO для s3 тестов.

Сценарии:
1) `health` ok
2) `tab_open` → `navigate` на `data:text/html,...` → `text` содержит ожидаемую строку
3) `snapshot` + `action click` по ref на странице с кнопкой, которая меняет DOM → `observe_changes(diff=true)` возвращает изменения
4) `tab_lock`:
   - agentA lock ok
   - agentB lock fail
   - после expiry или unlock agentB lock ok
5) `screenshot base64` возвращает валидный JPEG header
6) `screenshot s3` возвращает URL (MinIO)

### 3.3 E2E tests (MCP client ↔ wrapper ↔ Pinchtab)
- Используем MCP client harness (SDK) и вызываем tools так, как их будет дергать opencode.
- Проверяем:
  - инструменты объявлены (list tools),
  - основные happy-path сценарии,
  - корректные сообщения об ошибках,
  - отсутствие логов в stdout (только stderr).

## 4) Конфигурация и её настройка

### 4.1 Конфигурация wrapper (env)
- `PINCHTAB_MODE`: `auto` | `docker` | `external` (default: `auto`)
- `PINCHTAB_URL`: например `http://127.0.0.1:9867` (для external)
- `PINCHTAB_TOKEN`: bearer token (обязателен для external; для docker auto может генерироваться)
- `PINCHTAB_DOCKER_IMAGE`: default pinned (например `pinchtab/pinchtab:v0.4.0`)
- `PINCHTAB_DOCKER_NAME`: default `pinchtab`
- `PINCHTAB_STATE_DIR`: host path для volume (default `~/.pinchtab-mcp/state`)
- `PINCHTAB_PORT`: default `9867` (внутри)
- `PINCHTAB_PUBLISH`: `false|true` (если true — можно публиковать порт; по умолчанию false или loopback-only)

Token-saver defaults:
- `DEFAULT_SNAPSHOT_FORMAT`: `compact`
- `DEFAULT_MAX_TOKENS`: `2500`
- `DEFAULT_FILTER_INTERACTIVE`: `true`

Screenshots:
- `SCREENSHOT_DEFAULT_DELIVERY`: `base64|s3`
- `S3_ENDPOINT`, `S3_REGION`, `S3_BUCKET`
- `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`
- `S3_PREFIX` (например `pinchtab/`)
- `S3_PUBLIC_URL_BASE` (если нужен красивый публичный URL)

Логи:
- `LOG_LEVEL`: `debug|info|warn|error` (stderr only)

### 4.2 Конфигурация Pinchtab (env / config.json)
Wrapper должен уметь задавать (при docker-mode) ключевые env Pinchtab:
- `BRIDGE_PORT`, `BRIDGE_TOKEN`, `BRIDGE_HEADLESS=true`
- `BRIDGE_STATE_DIR=/data`, `BRIDGE_PROFILE=/data/chrome-profile`
- `BRIDGE_STEALTH=light|full`
- `BRIDGE_BLOCK_IMAGES`, `BRIDGE_BLOCK_MEDIA`
- `BRIDGE_NO_ANIMATIONS`
- `BRIDGE_TIMEOUT`, `BRIDGE_NAV_TIMEOUT`
- `CHROME_BINARY`, `CHROME_FLAGS`, `CDP_URL` (опционально)

## 5) Сборка pinchtab и pinchtab-mcp-wrapper

### 5.1 Pinchtab (как зависимость)
Рекомендуемый runtime (для VM без Chrome):
- Использовать официальный docker image `pinchtab/pinchtab:<tag>`.

Опционально, сборка из исходников (для dev-машин или CI):
- Требуется Go (версия как в upstream) и установленный Chrome/Chromium для запуска.

Команды (пример):
- `git clone https://github.com/pinchtab/pinchtab.git`
- `cd pinchtab && go build -o pinchtab .`
- `BRIDGE_HEADLESS=true ./pinchtab`

Docker build (если нужно):
- multi-stage build как в upstream Dockerfile (build stage + alpine runtime с chromium).

### 5.2 pinchtab-mcp-wrapper
Выбранная реализация: TypeScript/Node (официальный MCP SDK), stdio server.

Рекомендуемые targets:
- `make build` → компиляция TS в `dist/`
- `make test` → unit tests
- `make integration` → docker-compose + integration tests
- `make e2e` → MCP harness e2e

Docker distribution:
- `Dockerfile` для wrapper (node runtime)
- `docker-compose.yml`:
  - service `pinchtab` (volume state, без публикации порта наружу)
  - service `wrapper` (stdio обычно используется без портов; для локального dev можно добавить inspector hooks)

### 5.3 Рекомендованный UX "самый удобный для пользователя"
- По умолчанию: **одна команда** поднимает всё:
  - `docker compose up -d pinchtab`
  - `npx pinchtab-mcp-wrapper` (или `node dist/index.js`)
- Для полностью "одним кликом": опциональный `docker compose up` поднимает pinchtab и wrapper (но wrapper всё равно stdio и обычно запускается клиентом MCP; поэтому "always-on wrapper container" не делаем обязательным).

## Consequences / Trade-offs
- Docker sidecar упрощает окружение, но добавляет зависимость от Docker daemon.
- Сильная ставка на token-saver макро-инструменты требует аккуратного выбора дефолтов (`maxTokens`, `format`).
- S3 режим добавляет секреты (credentials) и инфраструктуру; поэтому base64 должен оставаться доступным.

## Open Questions
- Как именно opencode идентифицирует "агента" (owner) для lock: вручную задаётся вызывающей стороной или берём из session metadata MCP client?
- Нужен ли отдельный механизм "policy guardrails" поверх полного контроля браузера (например allowlist доменов) — сейчас не требуется.
